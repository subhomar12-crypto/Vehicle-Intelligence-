# PREDICT Vehicle Intelligence Platform

Unified FastAPI + PySide6 + PostgreSQL + Redis + SQLAlchemy 2.0 (async) vehicle intelligence platform.
Dual-mode: Headless (Windows service via NSSM, 24/7) + Desktop (GUI + embedded server).

## Project Context

This is a Python-based vehicle platform project with multiple phases (1-12+). Code is sometimes generated by external AI tools (e.g., Kimi) and needs systematic review for security vulnerabilities, async/await correctness, and datetime handling. Always flag these patterns proactively during any code review or file read.

When reviewing generated code:
- Always check for unsafe deserialization and replace with safe alternatives like json.loads()
- Verify all async/await patterns are consistent
- Ensure all timestamps use `time.time()` (float) ŌĆö never `datetime.now()` or `datetime.utcnow()`
- All DB timestamp columns use `sa.Float()`, never `sa.DateTime()`
- All ORM models use `Mapped[type]` + `mapped_column()`, never `Column()`
- All imports must be at the top of the file, never inside function bodies
- Production code uses `logging.getLogger(__name__)`, never `print()`
- GUI uses PySide6 only, never PyQt5/PyQt6
- No hardcoded paths ŌĆö use `get_config()`
- Review phase-by-phase if the codebase is organized that way

- If an API call or Task agent fails with a 500 error, do NOT retry the same approach more than twice. Instead, summarize what was attempted, the error, and suggest a manual debugging step the user can try.

## Architecture Rules (Non-Negotiable)

| Rule | Correct | Wrong |
|------|---------|-------|
| Timestamps | `time.time()` returns float | `datetime.now()`, `datetime.utcnow()` |
| DB timestamp columns | `sa.Float()` | `sa.DateTime()` |
| ORM style | `Mapped[type]` + `mapped_column()` | `Column()` |
| Imports | ALL at top of file | `import X` inside functions |
| GUI framework | `PySide6` | `PyQt5`, `PyQt6` |
| Logging | `logging.getLogger(__name__)` | `print()` in production code |
| Paths | `get_config().DATA_DIR` | Hardcoded `C:\...` paths |
| Parsing untrusted data | `json.loads()` | `eval()`, `exec()` |
| Async DB | `await session.execute()` | Sync `session.query()` |
| Repository pattern | `BaseRepository[ModelT]` | Raw SQL in routers |

## Common Bugs Found During Review (Watch For These)

These bugs were caught across 12 phases. Watch for them in any new or modified code:

1. **`eval()` on user data** ŌĆö Found in fleet.py parsing vehicle_ids. Always use `json.loads()`.
2. **Sync methods calling async functions without await** ŌĆö Found in ai_bridge.py. Returns coroutine objects instead of results. Ensure `async def` + `await` are paired.
3. **Non-existent method calls** ŌĆö Found in ai_bridge.py calling `quick_health_check()` which didn't exist. Verify methods exist on the target class.
4. **FileHandler before directory exists** ŌĆö Found in headless.py. `logging.FileHandler()` at module level crashes if `data/logs/` doesn't exist. Create directory BEFORE the handler.
5. **`import math` at END of file** ŌĆö Found in guardian_service.py. Function using math was defined above the import. NameError at runtime.
6. **Unused datetime imports** ŌĆö Found in validation.py and profiles.py. `from datetime import datetime` imported but never used. Remove dead imports.
7. **Stale phase comments** ŌĆö Comments like "Phase 10 will implement this" that are now done. Remove them.
8. **Hardcoded paths in docstrings** ŌĆö Found `C:\OBDserver\...` in billing_service.py docstring. No hardcoded paths anywhere.

## Review Protocol (For AI-Generated Code)

When reviewing code generated by external AI tools (Kimi, etc.):

1. **Grep scan first** ŌĆö Quick automated checks:
   - `datetime.now|datetime.utcnow` ŌåÆ must be zero
   - `eval\(|exec\(` ŌåÆ must be zero
   - `print\(` in `predict/core/` ŌåÆ must be zero (OK in `scripts/`)
   - `PyQt5|PyQt6` ŌåÆ must be zero
   - `Column\(` in models ŌåÆ must be zero

2. **Read critical files** ŌĆö Focus on:
   - Files that handle user input (routers, middleware)
   - Files with async code (services, AI modules)
   - Files that touch the database (repositories, services)

3. **Fix and report** ŌĆö For each issue:
   - State the file, line, and what's wrong
   - Apply the fix
   - Verify the fix doesn't break anything
